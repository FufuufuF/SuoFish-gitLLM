# 消息列表 — 分支感知 UI 设计文档

> 本文件记录消息列表在"分支/合并"场景下的 UI 设计决策，指导后续开发实现。

---

## 一、核心问题与设计目标

`context-messages` 接口会在当前线程的消息列表中**同时返回**：

1. **祖先线程（ancestor threads）的消息** — 当前分支的"背景上下文"，是从父线程继承下来的消息。
2. **当前线程自己的消息** — 用户在这条分支线上新产生的对话。
3. **学习简报消息（BRIEF）** — 合并时由 AI 生成、写入父线程的结构化摘要。

用户需要清晰感知：

- 「哪里发生了分叉」（分支起点）
- 「哪些消息是继承来的背景」（祖先消息区）
- 「哪些消息是我在这条分支上新说的」（当前分支消息区）
- 「这条简报是从哪个分支合并过来的」（BRIEF 标记）

---

## 二、消息类型与数据字段

每条 `ContextMessage` 携带 `threadId`（所属线程）和 `type`（消息类型）：

| 字段               | 含义                                                             |
| ------------------ | ---------------------------------------------------------------- |
| `threadId`         | 消息属于哪条线程（与当前活跃 threadId 对比即可区分祖先 vs 当前） |
| `type = 1 (CHAT)`  | 普通对话消息                                                     |
| `type = 2 (BRIEF)` | 学习简报消息（合并产物）                                         |
| `role = 1`         | 用户消息                                                         |
| `role = 2`         | AI 消息                                                          |
| `role = 3`         | 系统消息                                                         |

**判断规则（前端）**：

- `message.threadId !== activeThreadId` → 祖先线程消息
- `message.threadId === activeThreadId` → 当前分支消息
- `message.type === 2` → 学习简报，不论所属线程，都走专属渲染

## 三、UI 方案决策

### 3.1 方案对比

| 方案                   | 描述                                                                                                                            | 优点                                     | 缺点                                                       |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- | ---------------------------------------------------------- |
| A — 降低透明度         | 祖先消息 opacity 降低（如 0.5）                                                                                                 | 实现最简单                               | 对比度差，可读性受损；用户会误以为是 loading/disabled 状态 |
| B — 背景色分区         | 祖先区加浅色背景色块                                                                                                            | 区域感强                                 | 视觉重，滚动时感觉"杂乱"；对深色主题适配困难               |
| C — 分隔栏 + 轻量化 ✅ | 在分支起点处插入醒目的**「分叉点分隔栏」**，祖先消息整体减轻视觉权重（字号不变，仅降低文字颜色/图标亮度），当前分支消息正常显示 | 语义最清晰；不影响可读性；对深色主题友好 | 需要插入额外分隔元素                                       |

**决策：采用方案 C（分隔栏 + 轻量化）**

理由：与 git 实际体验对照，"从哪里分叉出来的"这件事最重要，一条醒目的分隔栏能让用户一眼理解空间结构，同时祖先消息保持可读性（仅降权、不隐藏）。

---

### 3.2 视觉分区方案（最终方案）

消息列表从上到下分为三段：

```
┌─────────────────────────────────────────┐
│  [祖先线程消息区]                        │
│  ┄ 主线 > Docker 探究                   │  ← 略微降权（text.secondary 颜色）
│  ┄ AI 头像灰色，消息内容颜色降级        │
│                                         │
│ ── [分叉点分隔栏] ──────── ✂ ──────── │  ← 醒目分隔线（见 3.3）
│     从「主线」第 7 条消息切出           │
│                                         │
│  [当前线程消息区]                        │
│  ● 正常全彩显示（同现有样式）           │
└─────────────────────────────────────────┘
```

**BRIEF 消息**在任何位置出现时，都用独立卡片样式覆盖（见 3.5）。

---

### 3.3 分叉点分隔栏（ThreadForkDivider）

插入时机：在当前线程的**第一条消息**之前插入。

```
┄ ┄ ┄ ┄ ┄ ┄ ┄ ┄ ✂ 从这里创建分支 ✂ ┄ ┄ ┄ ┄ ┄ ┄ ┄
                  Docker 部署探究
```

视觉规格：

- 水平方向：左右各一条虚线（`border-top: 1px dashed`），中间文字
- 文字内容：`从「{父线程标题}」切出` 或仅 `分支起点`（父线程标题由面包屑数据提供）
- 颜色：`primary.main`（品牌色），轻量 — 不用实线，避免过重
- 图标：剪刀（✂）或 `CallSplit` MUI 图标

**不显示分隔栏的情况**：

- 当前线程是主线（`threadType === 1`），没有祖先消息，无需分隔栏

---

### 3.4 祖先线程消息的轻量化规则

**不改变字号和气泡布局**，仅改变颜色权重：

| 元素                 | 正常状态                                 | 祖先消息状态                                       |
| -------------------- | ---------------------------------------- | -------------------------------------------------- |
| AI 头像              | `primary.main` 背景                      | `action.disabled` 背景（灰色）                     |
| AI 消息文字          | `text.primary`                           | `text.secondary`                                   |
| AI 底部工具栏        | 显示                                     | **隐藏**（不允许在祖先消息上重新生成）             |
| 用户气泡             | `action.hover` 背景，`text.primary` 文字 | `action.focus` 背景（略浅），`text.secondary` 文字 |
| 用户消息悬浮复制按钮 | 显示（hover 时）                         | **隐藏**（祖先消息只读感）                         |

> **核心原则**：祖先消息「可读但不可操作」，通过视觉降权而非隐藏内容来体现"背景"性质。

---

### 3.5 学习简报（BRIEF）消息的专属卡片

BRIEF 消息无论出现在哪条线程中，都渲染为醒目的卡片，与普通对话消息明显区分。

```
┌─────────────────────────────────────────────────┐
│ 📝  学习简报  ─  Docker 基础与安装              │  ← 标题栏（primary 色背景 stripe）
│  来源：探究分支「Docker 部署」                  │  ← 来源标注（可选，点击可跳转）
├─────────────────────────────────────────────────┤
│  ## 关键结论                                    │
│  - Docker 使用 cgroup + namespace 实现隔离...   │
│  ## 可执行步骤                                  │
│  - apt-get install docker.io                    │
│  ...                                            │  ← Markdown 渲染
│                                   [展开 / 收起] │
└─────────────────────────────────────────────────┘
```

视觉规格：

- 整体宽度：与 AI 消息等宽（不超过 85% 容器宽）
- 边框：左侧 3px `primary.main` accent 条
- 背景：`primary.main` 以 4% 透明度的叠色（`alpha(primary.main, 0.04)`）
- 标题栏：「📝 学习简报」badge + 来源分支标注
- 内容区：Markdown 渲染（复用 `MarkdownContent`），默认**折叠**显示前 3 行，点击「展开」查看完整
- 收起状态用渐变遮罩（`linear-gradient` 底部淡出）

**颜色语义**：BRIEF 是"带回来的知识"，用品牌主色 accent 而非警告色，传达"有价值"而非"有风险"。

## 四、MessageList 渲染逻辑

```
messages = [msg1(threadId=1), msg2(threadId=1), msg3(threadId=2), msg4(threadId=2)]
activeThreadId = 2

渲染步骤：
1. 找到第一条 threadId === activeThreadId 的消息索引 → firstCurrentIdx
2. firstCurrentIdx 之前的所有消息 → 祖先消息区（轻量化渲染）
3. 在 firstCurrentIdx 位置插入 <ThreadForkDivider>（仅当 firstCurrentIdx > 0 时）
4. firstCurrentIdx 及之后的消息 → 当前分支消息区（正常渲染）
5. 任意位置遇到 type === BRIEF → 切换为 <BriefMessageItem>
```

## 五、组件拆分

| 组件                | 文件                      | 职责                                        |
| ------------------- | ------------------------- | ------------------------------------------- |
| `MessageList`       | `message-list.tsx`        | 分区逻辑 + 遍历渲染，注入 `isAncestor` prop |
| `MessageItem`       | `message-item.tsx`        | 接收 `isAncestor?: boolean`，控制降权样式   |
| `ThreadForkDivider` | `thread-fork-divider.tsx` | 分叉点分隔栏，纯展示组件                    |
| `BriefMessageItem`  | `brief-message-item.tsx`  | 学习简报专属卡片                            |

### MessageItem 新增 Props

```typescript
interface MessageItemProps {
  message: Message;
  isAncestor?: boolean; // 是否祖先线程消息（影响降权样式）
  onCopy?: (content: string) => void;
  onRegenerate?: (messageId: string | number) => void;
}
```

### ThreadForkDivider Props

```typescript
interface ThreadForkDividerProps {
  parentThreadTitle?: string; // 父线程标题，用于显示「从「xxx」切出」
}
```

## 六、待确认问题

| #   | 问题                                                                                                   | 影响       | 建议默认                                                                                        |
| --- | ------------------------------------------------------------------------------------------------------ | ---------- | ----------------------------------------------------------------------------------------------- |
| 1   | **祖先消息工具栏是否全隐藏？** 还是只隐藏「重新生成」，保留「复制」？                                  | 体验细节   | 保留复制，隐藏重新生成（复制是读操作，合理；重新生成会改变上下文，不合理）                      |
| 2   | **BRIEF 默认展开还是折叠？** 首次出现时展开，后续滚动恢复时折叠？                                      | 信息密度   | 首次插入时展开；页面重新加载时折叠（用 localStorage 记忆 expanded 状态，以 messageId 为 key）   |
| 3   | **分叉点分隔栏是否需要显示「从哪条消息切出」的引用？** 即高亮父线程中的 `forkFromMessageId` 对应消息？ | 开发量中等 | MVP 不做，仅显示父线程名称；后续可做"高亮 + 跳转"交互                                           |
| 4   | **多级分支场景**：消息列表里可能有多段祖先（祖父、父、当前），如何分段？                               | 架构复杂度 | MVP 只区分「祖先 vs 当前」两段，不做多段分区；多级分支由 thread-branch-graph 侧边栏承担导航职责 |

### 问题确认

1. 保留[复制]
2. 默认折叠
3. MVP不做
4. 只区分`祖先`和`当前`

## 七、与其他组件的联动

- **面包屑栏（ThreadBreadcrumb）**：提供父线程标题文本，传给 `ThreadForkDivider`
- **Thread Branch Graph（侧边栏树）**：提供宏观结构视图，消息列表聚焦"当前线"的局部视图
- **MergeDrawer（合并 Drawer）**：生成 BRIEF 消息，写入父线程后触发父线程消息重新加载
